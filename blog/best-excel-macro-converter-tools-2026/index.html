Tier 1: Tools That Actually Preserve Macros

#1 — COM Automation via Python + xlwings or win32com

What it is: Driving Microsoft Excel itself via Windows Component Object Model (COM), using Python libraries like xlwings or win32com.client.

How macro preservation works: COM automation doesn't convert files in the SaaS sense — it instructs a real Excel instance to open the file and perform operations. Because Excel is the execution environment, macro code is not touched, not parsed, not re-encoded. It stays intact in the output format if the output format supports it (XLSM, XLSB, XLAM), or is explicitly preserved as a .bas export if converting to a non-macro format.

import win32com.client
import os

excel = win32com.client.Dispatch("Excel.Application")
excel.Visible = False


wb = excel.Workbooks.Open(r"C:\files\report.xlsm")


Export VBA modules before conversion

for component in wb.VBProject.VBComponents:
    if component.Type in [1, 2, 3]:  # Standard, Class, UserForm
        component.Export(rf"C:\backup\{component.Name}.bas")

Save as PDF

wb.ExportAsFixedFormat(0, r"C:\output\report.pdf")
wb.Close(False)
excel.Quit()

Limitations:

  • Windows only. COM is a Windows technology. Runs on Linux only in Wine, which introduces its own compatibility issues.
  • Requires a licensed Microsoft Office installation on the server.
  • Not horizontally scalable without significant infrastructure (multiple Windows VMs or container images with Office).
  • No multi-tenancy by default — one Excel process handles one file at a time.
Verdict: The gold standard for macro fidelity. Not viable as a cloud SaaS without significant infrastructure investment, but the correct solution for enterprise or internal tooling.

Cost: Free if you have Office licenses. The server/infrastructure cost is the real expense.

#2 — Aspose.Cells (and Aspose.Words for DOCM)

What it is: A commercial .NET/Java library from Aspose that implements its own Office file format parser — no dependency on Microsoft Office or LibreOffice.

How macro preservation works: Aspose parses the OOXML format directly, including the VBA project storage embedded within the file container. Because it implements the format spec rather than relying on an application to open the file, it can read, manipulate, and re-write macro code with documented API methods.

// Aspose.Cells - XLSM to XLSM with macro intact
Workbook workbook = new Workbook("input.xlsm");

// VBA project is accessible
VbaProject vbaProject = workbook.VbaProject;
foreach (VbaModule module in vbaProject.Modules)
{
    Console.WriteLine($"Module: {module.Name}");
    Console.WriteLine(module.Codes); // Full VBA source
}


// Save with macros preserved
workbook.Save("output.xlsm", SaveFormat.Xlsm);

For cross-format conversion (XLSM to PDF), Aspose renders the visible content to PDF and can optionally export VBA modules as separate files for preservation.

Limitations:

  • Price: $1,499 per developer license for Aspose.Cells. Enterprise pricing for Aspose.Total (all Office formats) starts at $3,000+.
  • Macro execution is not supported — Aspose reads and writes VBA code but cannot run it. If your workflow depends on macros executing during conversion, COM automation is required.
  • The Java version (Aspose.Cells for Java) exists but is less complete for VBA operations than the .NET version.
Verdict: The best option for building a commercial file conversion product that handles macros as a first-class feature. The license cost is the entry price for genuine macro fidelity in a cross-platform deployment.

Cost: $1,499/developer + $600/year maintenance.

Tier 2: Partial Macro Handling

#3 — LibreOffice Headless + Custom VBA Extractor

What it is: Running LibreOffice without a GUI on a Linux server, combined with custom code to extract VBA modules before conversion and optionally re-embed them.

How partial macro preservation works: LibreOffice opens XLSM/DOCM files and can convert them to other formats. It does not preserve macros across format conversions. However, the XLSM format is a ZIP container — you can unzip it, extract the vbaProject.bin binary, convert the document to XLSX with LibreOffice, and then use a library like oletools or olevba to parse the VBA binary and export module source.

This is a two-step extraction, not true preservation: you end up with the converted file (macros stripped) plus the VBA source code in separate .bas files. Whether that's acceptable depends on the use case.

Extract VBA source from an XLSM

cp report.xlsm report.zip
unzip report.zip -d extracted/
python3 -m olevba extracted/xl/vbaProject.bin > macros.txt

Convert the document

libreoffice --headless --convert-to xlsx report.xlsm --outdir output/

Limitations:

  • Not true macro preservation — you get the source, not a functional output with running macros.
  • olevba parses the VBA binary format, but obfuscated macros, P-code-only modules, and certain ActiveX components are not reliably extracted.
  • No commercial support. Bug fixes depend on the LibreOffice project.
Verdict: The open-source path to a macro-aware converter. Appropriate for self-hosted tools where extraction (not execution) is the goal.

Cost: Free. Infrastructure is the only cost.

What the Rankings Reveal About the Market

FAQ

Is there any free online tool that preserves Excel macros? No. As of 2026, no free consumer converter (Smallpdf, ILovePDF, CloudConvert free tier, Zamzar, OnlineConvertFree) preserves VBA macros across format conversions. The technical barrier — executing or faithfully re-encoding VBA at scale — has not been solved in the consumer tier.

Can LibreOffice preserve macros when converting? LibreOffice preserves macros when the output format supports them (e.g., ODS with Basic macros). For XLSM to PDF or XLSM to XLSX conversion, the VBA code is dropped. LibreOffice does not implement Microsoft's VBA runtime and cannot guarantee fidelity for complex macro operations.

What's the cheapest way to get macro preservation? COM automation via xlwings or win32com is free if you have an Office license. This is Windows-only and requires a licensed Excel installation on the machine running the conversion.

Is Aspose.Cells worth the price? For a commercial product where macro fidelity is a paid feature, yes. At $1,499/developer, it prices itself out of personal projects but is reasonable for a SaaS charging $10-20/month to business users with complex Excel automation files.
